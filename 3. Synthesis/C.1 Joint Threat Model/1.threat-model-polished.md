## 1. Purpose

Workstreams A and B each examined Meridian FinServe from a single surface — network forensics and web application security respectively. Neither document alone shows how those two surfaces connect into a single, cohesive risk picture. **This workstream does.**

This document presents a STRIDE-based threat model treating Meridian FinServe as **one integrated system**, mapping threats from both prior workstreams against specific architectural components with special focus on attack chains that cross both surfaces simultaneously.

---

## 2. Architecture Overview (Baseline)

**See `architecture/1.before.mermaid`** for the complete current-state ("As-Is") network architecture used as the baseline for this threat model.

**Client Profile:** Meridian FinServe Pvt. Ltd. is a fictional mid-sized Indian non-banking financial company headquartered in Mumbai with presence across nine cities. The company offers SMB lending, merchant payments, and an embedded credit product. Roughly 720 employees support a customer base of approximately 180,000 borrowers and 22,000 merchants through a customer-facing portal and a partner portal.

**Key In-Scope Components:**
- Customer Portal (lending applications, EMI servicing, statements)
- Partner Portal (merchant onboarding and reconciliation)
- Application Servers, Auth/IAM service, Loan Processing Engine, Payment Gateway
- Core Database, Active Directory, SIEM, Backup Storage (DC1/DC2)
- Branch offices (9 cities) connected via VPN
- Public cloud footprint (S3 storage, API integrations)

## 3. STRIDE — Concept Note

**STRIDE** is a threat categorisation framework developed at Microsoft (1999). It asks one structured question per category for each system component, producing an actionable threat register.

| Category | Security Property Violated | What It Means |
|---|---|---|
| **S**poofing | Authentication | Attacker impersonates a legitimate user or system |
| **T**ampering | Integrity | Data or code modified without authorisation |
| **R**epudiation | Non-repudiation | Actions taken that cannot be attributed or logged |
| **I**nformation Disclosure | Confidentiality | Data accessed by unauthorised parties |
| **D**enial of Service | Availability | Legitimate users blocked from the system |
| **E**levation of Privilege | Authorisation | Entity gains capabilities beyond what was granted |

# 7. Cross-Surface Attack Chains (C.1 Requirement)

## Chain 1: Web Compromise Leads to Network Beaconing (Web → Network)

**Direction:** WS-B → WS-A

### Attack Flow

1. Attacker exploits SQL Injection (A03) or IDOR (A01) on Customer Portal.
2. Extracts session tokens or staff credentials.
3. Gains foothold on Application Server.
4. Deploys Trickbot or Cobalt Strike.
5. Anomalous C2 beaconing becomes visible in network traffic (WS-A).

**STRIDE Path:** I → S → E → T → I

**Why this matters:**
The initial compromise occurs in the web application, but the first observable evidence may appear at the network layer through beaconing and lateral movement activity.

---

## Chain 2: Network Foothold Enables Deep Web Data Theft (Network → Web)

**Direction:** WS-A → WS-B

### Attack Flow

1. Trickbot lands on branch workstation.
2. LSASS dump performed.
3. Pass-the-Hash used for lateral movement.
4. Application Server compromised.
5. DB credentials or portal admin tokens extracted.
6. Customer Portal abused for large-scale SQLi or IDOR data theft.

**STRIDE Path:** I → S → E → I

**Why this matters:**
A network compromise directly enables privileged access to the web environment and its sensitive borrower data.

---

## Chain 3: Parallel Attack — Network Assists Web Layer

**Direction:** WS-A ∥ WS-B

### Attack Flow

1. Trickbot infects branch workstation through phishing.
2. Establishes command-and-control beacon.
3. Performs lateral movement.
4. Extracts JWT secrets, DB credentials, and AWS keys from Application Server.
5. Uses stolen secrets to forge admin sessions.
6. Performs mass IDOR extraction.
7. Accesses cloud storage using stolen AWS credentials.

**STRIDE Path:** S → I → E → S → I

**Surface Crossing:**
Network-level filesystem access provides secrets that directly enable privileged web-layer compromise.

## Top 5 Immediate Actions

1. Parameterised queries (blocks SQLi and Chain 2).
2. HTTPS enforcement with HSTS (reduces token theft risk in Chain 1).
3. Network microsegmentation (limits lateral movement).
4. Secrets vault implementation (breaks Chain 3).
5. WAF deployment with strong server-side authorisation controls.

---

# 9. Mitigations (Chain Blockers Highlighted)

## TB-1 — Internet to DMZ

* HTTPS everywhere with HSTS (**Chain Blocker — Chain 1**)
* WAF with OWASP Core Rule Set (**Chain Blocker — Chain 2**)
* Parameterised database queries
* MFA for all portal users

## TB-2 — DMZ to Internal

* Restrict DMZ-to-DB connectivity
* File upload validation
* Object-level authorisation controls to eliminate IDOR

## TB-3 — Internal to Branch / Cloud

* Network microsegmentation (**Chain Blocker — Multiple Chains**)
* EDR with behavioural C2 detection
* Privileged Access Management (PAM)

## Application & Data

* Secrets Vault (**Chain Blocker — Chain 3**)
* Immutable audit logging
* Regular vulnerability and patch management

# 11. Analyst Note

The web application and network environments are not independent risk domains. Each surface can enable, amplify, or conceal attacks occurring on the other.

The three cross-surface attack chains demonstrate that:

* Web compromise can create observable network-layer indicators.
* Network compromise can enable deeper web-layer abuse.
* Shared secrets and credentials form critical trust relationships between both environments.

Implementing the four primary controls below would disrupt every major attack chain identified in this assessment:

1. HTTPS enforcement
2. Parameterised queries
3. Network microsegmentation
4. Centralised secrets vault
